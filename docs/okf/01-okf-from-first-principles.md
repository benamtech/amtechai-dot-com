# OKF From First Principles

> Sources: the OKF v0.1 spec (`okf/SPEC.md`), Google Cloud's launch blog ("How the Open Knowledge Format can improve data sharing"), the `knowledge-catalog` repo tree, and a real sample concept (`okf/bundles/stackoverflow/tables/posts_questions.md`). All fetched June 18, 2026. Running research notes are interleaved as `> Note:` blocks — these are deliberate; they feed the future article.

## 1. What OKF is, in one breath

OKF is **an open, vendor-neutral format for representing knowledge as a directory of markdown files with YAML frontmatter.** That is the entire format. There is no schema registry, no central authority, no required SDK, no runtime. A bundle is readable by a human in any text editor, parseable by an agent without bespoke tooling, diffable in git, and portable across tools, organizations, and time.

Google's framing: knowledge — "the metadata, context, and curated insight that surrounds data and systems" — is *"authored by people, generated by agents, exchanged across organizations, and consumed by both."* OKF is the lingua franca for that exchange.

> Note (thought process): The first instinct is to assume a Google "knowledge format" means RDF/JSON-LD/ontology heavy machinery. It is the opposite. OKF is deliberately *less* structured than schema.org. The blog never even mentions RDF. This is a bet that **markdown + a single required field beats a rich ontology** because adoption — "how many parties speak it" — is the only thing that gives a format value. That is the same reason `AGENTS.md`/`CLAUDE.md` won as conventions: low ceremony, high reach. The blog explicitly cites the `AGENTS.md`/`CLAUDE.md` pattern and Karpathy's "LLM wiki" gist as ancestry.

## 2. The data model, from the ground up

### 2.1 The concept document — the only real object

Everything in OKF is a **concept**: one UTF-8 markdown file describing one thing. The "thing" can be a tangible asset (a database table, an API endpoint) or an abstract idea (a metric, a process, a playbook, a reference). Structure:

```
---
[YAML frontmatter]
---
[Markdown body]
```

**Frontmatter fields:**

| Field | Status | Meaning |
| --- | --- | --- |
| `type` | **REQUIRED** | A short free string naming the kind of concept (`BigQuery Table`, `Metric`, `Playbook`, `Reference`…). **Not registered centrally** — you invent the vocabulary. |
| `title` | recommended | Display name. If absent, derive from filename. |
| `description` | recommended | One sentence. Used for index snippets, search previews. |
| `resource` | recommended | A URI uniquely identifying the underlying asset. Absent for purely abstract concepts. |
| `tags` | recommended | YAML list of short strings for cross-cutting grouping. |
| `timestamp` | recommended | ISO 8601 datetime of last meaningful change. |
| *(any key)* | optional | Producers MAY add arbitrary keys. Consumers MUST preserve unknown keys and MUST NOT reject documents for having them. |

The **concept ID** is implicit: the file path within the bundle minus `.md`. `tables/orders.md` → concept ID `tables/orders`. No GUIDs, no manifest of IDs — the filesystem *is* the namespace.

> Note: One required field. That is the whole conformance surface for content. The genius and the risk are the same: it will accept almost anything, so quality is entirely a producer discipline, not a format guarantee. This maps exactly onto our own "information-gain is the publishing bar" rule — OKF won't enforce quality, *we* must.

### 2.2 The body — structural markdown, with conventional sections

The body is plain markdown, but producers SHOULD favor *structural* markdown — headings, lists, tables, fenced code — so both humans and parsers can navigate it. Three section headings carry **conventional meaning** (not required, but understood by consumers):

- `# Schema` — structured field/column description, usually a markdown table.
- `# Examples` — concrete usage, usually fenced code blocks.
- `# Citations` — external sources backing claims, as a numbered list of markdown links. Citation targets may be absolute URLs, bundle-relative paths, or paths into a `references/` subdirectory.

### 2.3 Relationships — links, with meaning in the prose

This is the most important and most counterintuitive design choice. **Relationships are ordinary markdown links between concept files.** A link from concept A to concept B asserts *a* relationship. The *kind* of relationship (parent/child, references, joins-with, depends-on) is conveyed by **the surrounding prose, not by the link itself.** There is no typed-edge syntax.

Two link forms: bundle-relative (begins with `/`, recommended) and relative (`./concept.md`). Consumers **MUST tolerate broken links** — a link to a missing target is not an error.

> Note (this is the unlock for us): Our article system already stores `internalLinks: [{ href, label, reason }]`. The `reason` field is *literally* "the relationship conveyed by surrounding prose." We invented OKF's edge semantics before OKF existed, because entity-SEO forced us to. That is not a coincidence; it is the same problem (make relationships legible to a machine that fans out across a graph) solved by the same means.

### 2.4 The bundle — a directory, optionally a git repo

A **bundle** is the directory tree of concepts. Distribution forms: a git repo (recommended — gives history, attribution, diffs), a tarball/zip, or a subdirectory of a larger repo. Hierarchy (subdirectories) expresses implicit parent/child relationships.

Two **reserved filenames**, both optional:

- `index.md` — a directory listing for *progressive disclosure*. No frontmatter; body is markdown sections of `* [Title](url) - description` bullets. May appear in any directory including root. Producers MAY generate it; consumers MAY synthesize it on the fly.
- `log.md` — an update history for that directory scope. Date-grouped (`## YYYY-MM-DD`), newest first, entries like `* **Creation**: Established [concept](/path.md).`

The bundle root MAY declare the spec version with `okf_version: "0.1"` in its `index.md` frontmatter.

> Note: "Progressive disclosure" is the agent-facing point of `index.md`. An agent doesn't load 400 files; it reads `index.md`, picks the relevant branch, descends. This is the OKF equivalent of our `/articles` → "Browse by topic" → `/articles/all` funnel, and of `codegraph.json` for the repo. Same instinct: give the reader (human or agent) a cheap map before the territory.

### 2.5 Conformance and the permissive consumption model

A bundle is **conformant** if: (1) every non-reserved `.md` has a parseable YAML frontmatter block; (2) every frontmatter block has a non-empty `type`; (3) reserved files follow their structure when present. That's it.

Consumers **MUST NOT** reject a bundle for: missing optional fields, unknown `type` values, unknown extra keys, broken links, or missing `index.md`. This permissiveness is intentional — bundles "grow, get refactored, and are partially generated by agents," and the format has to survive all of that.

Versioning is `<major>.<minor>`: minor = backward-compatible additions (new optional fields, new conventional headings); major = breaking changes (renamed required fields, changed reserved filenames).

### 2.6 Explicit non-goals

OKF deliberately does **not**: define a fixed taxonomy of concept types; prescribe storage/serving/query infrastructure; or replace domain schemas (Avro, Protobuf, OpenAPI). It *references* those; it does not subsume them. It is a format, not a platform.

## 3. The reference implementation (what Google shipped)

The `knowledge-catalog` repo is a working demonstration, useful as a model for our own tooling:

- **Enrichment agent** (`okf/src/enrichment_agent/`, Python): walks a source (e.g. BigQuery), drafts an OKF concept per table/view in pass one, then an LLM enriches with web-sourced context and citations in pass two. CLI: `python -m enrichment_agent enrich --source bq --dataset <p>.<d> --out ./bundles/<name>`.
- **Visualizer** (`viz.html`, one per bundle): a self-contained static HTML file that renders the bundle as a force-directed graph — color-coded by `type`, with detail panels, backlinks, and search. No backend. CLI: `... visualize --bundle ./bundles/<name>`.
- **Sample bundles**: `crypto_bitcoin`, `ga4`, `stackoverflow` — each a `datasets/` + `tables/` (+ `references/`) tree with `index.md` and a generated `viz.html`.
- **`mdcode`** (TypeScript): a separate tooling implementation.

A real concept from the StackOverflow bundle (`tables/posts_questions.md`) frontmatter:

```yaml
---
type: BigQuery Table
resource: https://bigquery.googleapis.com/v2/projects/bigquery-public-data/datasets/stackoverflow/tables/posts_questions
title: Stack Overflow Questions
description: This table contains information about all question posts on Stack Overflow...
tags: [Stack Overflow, questions, posts, programming, community, schema, data dump]
timestamp: '2026-05-28T23:31:54+00:00'
---
# Schema  …table of columns…
# Common query patterns  …SQL fenced blocks…
# Citations  …numbered links…
```

> Note: Two facts matter for us. (1) The reference producer is an *agent*, and the reference consumer renders a *graph visualization from static files with no backend* — exactly the two ends we care about (agent authorship, zero-infra agent/crawler consumption). (2) `resource` points at the canonical machine address of the thing. For us the analog is the live article URL — which ties the OKF concept back to the indexed, JSON-LD-bearing web page. OKF and classic SEO are not either/or; the `resource` link is the seam between them.

## 4. OKF through an SEO lens

### 4.1 Traditional SEO

Classic SEO still sets the floor: crawlable pages, helpful content, accurate titles, fast render, clean links, source attribution. Google has said AI features in Search are rooted in the core ranking/quality systems. OKF does **nothing** for traditional SEO directly — a bundle of markdown in a repo is not your indexed website. Its value is upstream of and adjacent to the indexed site:

- Producing an OKF bundle forces the discipline (entities named, relationships made explicit, claims cited) that *also* produces better on-page content and JSON-LD.
- The bundle is a clean, link-dense, JS-free corpus — ideal for any crawler or retrieval system that prefers text over a rendered SPA.

### 4.2 AI / agent SEO

This is where OKF lives. Three current realities:

1. **Query fan-out.** AI Overviews and AI Mode issue multiple sub-searches across subtopics and sources. Pages that define entities clearly and sit inside a topical graph win; isolated keyword pages lose. OKF *is* a topical graph.
2. **Entities over keywords.** Retrieval and grounding work on things and their relationships. A concept-per-entity bundle is the most direct possible expression of "what this site is about, and how its ideas connect."
3. **Agents consume context, not pages.** "As foundation models improve, the lack of relevant context often limits what they can do." An OKF bundle is context an agent can ingest wholesale — read `index.md`, fan out, follow typed-by-prose links, respect citations.

> Note (the synthesis): Our three doctrines collapse into OKF cleanly. **Entity SEO** = one concept document per entity. **Information-gain SEO** = the publishing bar that OKF won't enforce for us, so we keep enforcing it (a concept that restates an existing concept is noise in a graph designed for de-duplicated, densely-linked knowledge). **Agent-first development** = OKF's literal premise ("authored by people, generated by agents, consumed by both"). We have been running the *repo* this way for months — `AGENTS.md`, `codegraph.json`, `wiki/`, `docs/memory/` are an informal internal OKF bundle. The blog cites `AGENTS.md` as the ancestral pattern. So adopting OKF for the *public* knowledge graph is not a new philosophy; it is applying our internal practice outward, in a format Google now lets the whole ecosystem read.

## 5. Why this is the right moment to also fix storage and prerendering

OKF surfaces a latent problem we already had: **our knowledge is trapped in two unfriendly substrates.**

- Articles live as TypeScript objects and hand-built TSX inside the app bundle. You cannot project TypeScript-in-source into markdown, JSON-LD feeds, or a bundle without parsing code.
- The site is a client-rendered SPA with exactly one HTML file. A crawler or agent that does not execute JavaScript sees an empty shell. There is no `sitemap.xml`, `robots.txt`, or `llms.txt`.

Both must be fixed for *any* agent-first outcome, OKF or not. OKF just makes the fix legible: the moment knowledge is **data in a store** rather than **code in a bundle**, projecting it into (a) React pages, (b) prerendered HTML, (c) JSON-LD, (d) an OKF bundle, and (e) sitemaps becomes mechanical. That projection architecture is the subject of [`03-adaptation-plan.md`](./03-adaptation-plan.md).
