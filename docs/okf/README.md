# AMTECH × Open Knowledge Format (OKF)

This folder is the research-and-adaptation record for bringing the AMTECH article system into conformance with Google's **Open Knowledge Format (OKF) v0.1**, released June 2026.

It exists for two reasons:

1. **Operational** — to decide and document exactly how we satisfy the standard without tearing down what already works.
2. **Editorial** — to seed a future AMTECH article about doing this from first principles. As we research, we capture findings, thought process, and actionable insights here so the eventual article is sourced, not reconstructed.

## The one-paragraph thesis

OKF is not a competitor to our knowledge-graph SEO doctrine — it is the **same philosophy in a different representation**. Entity-first SEO says "connect every page to the real-world things it is about." Information-gain SEO says "only publish what adds a distinction." Agent-first development says "write knowledge that humans curate and agents read and update." OKF v0.1 is precisely a portable file format for that: *a directory of markdown files with YAML frontmatter, authored by people, generated by agents, consumed by both.* Our article system is already OKF-shaped in its data model; what we lack is (a) a static, agent-readable surface (we are a JS-only SPA today) and (b) a canonical store we can project from. The plan turns our article data into **one canonical model with many projections**, of which an OKF bundle is one — added alongside the React site, never replacing it.

## Decisions captured here (so far)

- **Adopt OKF as a projection, not a rewrite.** Keep the React article experience. Emit an OKF bundle at build time from the same source of truth. (See plan, Phase 1.)
- **Treat entities as first-class concepts, not just tags.** Our knowledge graph already has two node kinds — articles and the entities they connect (services, problems, places, industries, use cases). In OKF, *both* become concept documents. This is the upgrade, not a constraint.
- **Fix the prerender gap regardless of OKF.** A JS-only SPA is invisible to non-executing crawlers and most agents. Prerendering articles + knowledge maps to static HTML, plus `sitemap.xml`/`robots.txt`/`llms.txt`, is required for agent-first SEO and is a prerequisite the OKF work rides on.
- **Move article content to Supabase — but phased.** Now that we have a database, articles should stop being TypeScript objects hand-edited in source. Phase the migration so it is never a big-bang cutover.

## Status (2026-06-18)

**Phases 1, 2, and 3 are built and green** (Phase 3 at the projection stage). Zero new dependencies beyond what shipped; nothing destructive; Node 24 runs the `.ts` build scripts directly.

Source & tooling:
- `src/lib/knowledge/` — the single authored source: `concepts.ts` (façade + block→markdown), `entities.ts` (registry), `articles/` (React-free `ArticleDefinition` data for all 8 published articles).
- `src/components/articles/useArticleHead.ts` — per-page title/description hook; article pages are now thin wrappers over imported data.
- `scripts/okf/` — `emit`, `build-okf`, `validate-okf`, `prerender`, `db-seed-sql`, `db-verify`.
- Commands: `okf:build` / `okf:validate` / `okf:validate:phase3` / `okf:check` / `okf:prerender` / `okf:db:seed-sql` / `okf:db:verify`.

Outputs:
- `public/okf/**` (**73 concepts** + 5 `index.md`) and `public/{sitemap.xml,robots.txt,llms.txt}`, copied to `dist/`.
- Prerendered static HTML for 8 article routes + `/articles` + `/articles/all` (postbuild), with per-page `<head>` + JSON-LD.
- Live Supabase tables `concepts` / `concept_edges` / `concept_citations` (73 / 166 / 19), published-only RLS, seeded from the façade.

Checks: `okf:check` → **0/0** at hard stage 2; `okf:validate:phase3` → G8/G9 pass; `okf:db:verify` → DB matches source, RLS exposes only the 8 published.

Arc of the work:
- **Phase 1** — conformance + agent-readable surface; the validator surfaced 22 orphan playbook nodes (linking rule written in prose, never in data).
- **Phase 2** — fixed that *by abstraction*: 26 entity concepts with edges derived from each node's `uc`/`city`/`subtype`. Zero orphans, bidirectional graph.
- **Phase 3** — consolidated article content into React-free data (the real keystone), prerendered routes, and projected the graph into Supabase as a queryable published surface. Architecture: **façade-as-source, DB-as-projection** (`06-phase-3-foundation.md`).
- **Remaining (optional):** the authoring cutover (build reads the DB) — gated on CI service-role creds and a reason to author outside code (the Phase-4 agent loop).
