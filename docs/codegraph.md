## Architecture at a glance

- **App type:** Vite + React 18 + TypeScript marketing/conversion website.
- **Runtime:** Static SPA served from `dist/`; Supabase provides database, storage, and Edge Functions. Netlify Functions provide the AI Employee claim front door. The repo also contains an `AI_EMPLOYEE_MVP/` product bundle for the Hermes/Twilio AI employee provisioning flow.
- **Build pipeline (`npm run build`):** `skills:build` materializes public skill packages, `okf:build` regenerates OKF/discovery files, `vite build` writes the SPA bundle, then `postbuild` (`scripts/okf/prerender.ts` + `scripts/seo/validate-seo.ts`) writes per-route static HTML for ALL 32 public routes (marketing, conversion, articles, hubs, skills) into `dist/` and fails the build if meta/body validation does not pass (real content + per-page `<title>`/description/canonical/OpenGraph/JSON-LD). The SPA boots with `createRoot().render()` (a client render, not hydrate), so it simply replaces the prerendered `#root` content — crawlers/agents get real content, users get the interactive app. Requires Node 24 (pinned via `netlify.toml` `NODE_VERSION` + `.nvmrc`) because the build scripts are `.ts` run directly via type stripping.
- **Knowledge / OKF surface:** the article knowledge graph projects to an Open Knowledge Format bundle (`public/okf/**`, 74 concepts) + `public/{sitemap.xml,robots.txt,llms.txt}`, all committed and copied into `dist/` by Vite. Single authored source = `src/lib/knowledge/` façade. Draft/research articles live under `docs/article-drafts/` until they are intentionally published into the live article graph. See `docs/okf/` and the "Knowledge graph / OKF" file-ownership entries below.
- **Per-page meta / SEO source of truth:** `src/lib/seo/pageMeta.ts` is the SINGLE authority for every public route's `<head>` (title, description, canonical, Open Graph, Twitter, robots, JSON-LD, agent-map) plus a real readable body for marketing/hub routes. It reads through the same façade boundary the OKF bundle + DB are projected from (`getConcepts()`, `articleDefinitions`, `src/lib/skills/registry.ts`) — never re-authoring concept data — so meta, OKF markdown, prerendered HTML, and the Supabase concept tables all project from one model (and inherit the future DB authoring cutover for free). `src/lib/seo/renderHead.ts` is the React-free string projection used by the prerenderer; `src/components/seo/SeoManager.tsx` applies the SAME `PageMeta` imperatively on every SPA route change (it replaced the old title-only `useArticleHead`, now a no-op shim). All 32 public routes are prerendered (was 13). Canonical-adjacent Markdown twins are emitted at `/articles/<slug>.md` (a "Knowledge map" traversal header from concept edges + the full body) and advertised via `<head>` `rel="alternate"`; `public/_headers` sets markdown content-type + caching. Validate with `npm run seo:validate` (gated in `postbuild`). Edge-based same-URL `Accept` negotiation is deferred (measure-then-decide) — see `docs/memory/plan-2026-06-19--unified-meta-and-delivery.md`.
- **Shareable agent skills surface:** `/skills` and `/skills/okf-audit` are live first-fetch skill consumption pages. Canonical source lives in `src/lib/skills/`; the public Git install source is `https://github.com/benamtech/amtech-skills-registry`. `scripts/skills/build-skills.ts` materializes complete skill folders (`SKILL.md`, `references/`, `assets/`, metadata; scripts supported by the format) into `public/skills/**` views: `use.md`, `agent.md`, manifest, file indexes, raw files, checksums, and zip archives. Each website page and manifest links branch and commit-pinned GitHub source; the domain authority file pins the matching repository commit/path and archive/file SHA-256 values. Start with `docs/UNIVERSAL_SKILL_LINK_CONTRACT.md`, `docs/SKILL_MATERIALIZATION_PIPELINE.md`, and `src/lib/skills/registry.ts`. The `/skills/:slug` page renders directly from the skill data model (not marketing copy): `scripts/skills/build-skills.ts` emits a React-free `src/lib/skills/generated/skill-content.ts` (use.md text + every file's content), and `src/lib/skills/renderSkillContent.ts` (+ `markdown.ts`) turns it into one HTML string consumed by BOTH the prerenderer and `src/pages/SkillDetail.tsx` (via `dangerouslySetInnerHTML`). The per-skill `use.md`/`agent.md` content (decision-tree task verb, Output Contract, inputs/outputs) is driven by the `taskVerb`/`inputs`/`outputContract`/`outputsSummary` fields on each `SkillDefinition` in `registry.ts` (never hardcoded), and these two agent-entry surfaces are **signed**: `certificate.bootstrap` binds their digests and the link-first verifier recomputes them (`BOOTSTRAP_DIGEST_MISMATCH`) — so crawlers, agents, and users get identical, content-rich pages (title/description/metadata, the use.md bootstrap, source/provenance links, use cases, every file with inline contents, download/machine views). **The `/skills` hub itself is self-bootstrapping the same way (M0 of the CA standard):** `src/lib/skills/renderHubContent.ts` is the shared React-free renderer (consumed by both `scripts/okf/prerender.ts` and the one-line `src/pages/Skills.tsx`) emitting the catalog-level agent-instruction block, the web/file/install decision tree, and an enumerated per-skill link list; `build-skills.ts` also emits the machine catalog `public/skills/catalog.json` (`amtech-skill-catalog/v1`) plus hub `use.md`/`agent.md`. `validate-skills.ts` enforces the G-M0 gates (catalog matches the registry; hub HTML carries the instruction block + decision tree). NOTE: at M0 the catalog carries structural fields only — verdict/trustTier/authoritySequence/checkedAt are earned by M1/M3/M4 and intentionally omitted.
- **Routing:** `src/App.tsx` owns all public routes. Most brand pages use `src/components/layout/Layout.tsx`; conversion flows (`/claim`, `/apply`, `/schedule-call`, `/pay`, etc.) render standalone. Current in-app site copy is the authoritative source after the June 2026 homepage, article-library, and AI Employee claim revisions.
- **Supabase client:** `src/lib/supabase.ts` reads `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY` and exports one browser client. If either var is missing it logs a console error and falls back to placeholder values instead of throwing at module load, so a missing local `.env` no longer white-screens the whole app (it is imported transitively by statically-loaded routes like `/apply`). Use the anon/publishable key only — never a service role key (every `VITE_*` value is inlined into the public bundle).
- **Database migrations:** `supabase/migrations/*.sql` define public form tables, intake storage references, booking availability and applications. `20260618210000_okf_concepts.sql` adds the OKF knowledge tables (`concepts`, `concept_edges`, `concept_citations`) with published-only public-read RLS — applied to the live project.
- **Edge Functions:** `supabase/functions/*/index.ts` handle Stripe PaymentIntent creation and Resend transactional emails.
- **AI employee bundle:** `AI_EMPLOYEE_MVP/` contains the Phase 2/3 build plan, Hermes profile template, onboarding schema, Twilio number/provisioning scripts, and the local provision hook used by the deployed Netlify claim function.

Trust-minimized (not trustless): witness co-signatures + external anchors are scaffolded but inactive. Spec: `docs/skills/standard/03` §"Transparency log — Option B". **Trust federation + broadcast (docs/skills/standard/11, 2026-06-22):** `scripts/signing/sign-anchor.ts` emits the **broadcast anchor** — an AMTECH Signed Artifact `status` certificate (reusing `amtech-signed-artifact/v2`, NOT a new format) whose subject is the STH (binds its SHA-256/SHA3-512, timestamped, Ed25519-signed; optional PGP co-sign via `AMTECH_ANCHOR_PGP_KEY`) — served at `/.well-known/authority/anchor/certificate.json(.sig)` and verifiable with the existing `artifact:verify`; gated by `validate-skills`; surfaced on `/registry`. **Federation:** the verifier resolves STH **witness** keys by their own `signingKeyUrl` (`ResourceLoader.keyByUrl`) so a witness can live on another domain, counting valid witness sigs against `{minWitnessSigs}`. **External anchoring** is a pluggable `ANCHOR_VERIFIERS` registry (empty "breaker" — register a `type` verifier when broadcasting STH roots to Bitcoin/Nockchain/etc.; chain-agnostic; `requireAnchor` fails closed until then). ## Agent usage notes

1. Start with this file and `wiki/db-forms-endpoints.md` before grepping.
2. For route changes, inspect `src/App.tsx`, the page file, then only the referenced component folder.
3. For data changes, inspect the relevant migration, frontend service/page submit handler, and Edge Function if present.
4. For design changes, read `AMTECH_STYLE_GUIDE.md`, `COST_CALCULATOR_STYLE.md`, and relevant notes in `wiki/design-notes.md`.
5. For article research/drafting, use `docs/skills/amtech-article-research-writer/`. For article publishing from supplied copy, use `docs/skills/amtech-article-publisher/` before editing article data, graph nodes, or OKF outputs. Use `docs/article-drafts/` for reviewable research/draft artifacts before publishing.
6. For shareable/free public agent skill work, start with `docs/UNIVERSAL_SKILL_LINK_CONTRACT.md`, `src/lib/skills/registry.ts`, and `scripts/skills/build-skills.ts`; for research context read `docs/AMTECH_SHAREABLE_SKILLS_STRATEGY.md`, `docs/SKILL_MATERIALIZATION_PIPELINE.md`, and `wiki/research/2026-06-19-shareable-agent-skills-and-projection-pipelines.md`.
7. Keep this codegraph updated whenever a route, table, endpoint, feature folder, article-system surface, or durable knowledge-graph/research source changes.
8. For AI Employee MVP work, start with `docs/AI_EMPLOYEE_MVP.md` and `AI_EMPLOYEE_MVP/BUILD-PLAN.md`; dry-run provisioning before any real Hermes/Twilio side effects.
